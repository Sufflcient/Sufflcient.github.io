    容器逃逸

    发表于 2026-04-12 分类于 Web

    当我们在对linux操作系统getshell之后,还需要判断该shell是否处于虚拟环境当中,并判断位于什么类型的虚拟环境当中,一般来说,shell的虚拟环境有虚拟主机、docker和Kubernetes(K8s)三种类型

    虚拟主机

    虚拟主机

    是一台物理服务器被分割成多个”虚拟空间”,用户共享CPU、空间等资源。

    • 其中多个网站共享同一台物理服务器的资源
    • 共享操作系统、Web服务器进程
    • 通过域名/目录来隔离不同的用户
    • 主要用于Web的托管服务

    特点:

    • 共享IP地址
    • 只能访问自己的Web目录(如 /var/www/sitea)
    • 没有独立的操作系统
    • 没有root权限

    虚拟机

    则是利用虚拟化技术将一台物理服务器划分成多个独立的”虚拟专用服务器”(一般指的是VPS或云服务器),每一个虚拟机都有它独立的操作系统。

    • 通过Hypervisor 虚拟化出完整的计算机系统
    • 每一个VM都有独立的操作系统
    • 完全隔离的硬件资源
    • 主要用于云服务器、VPS

    特点

    • 独立IP地址
    • 有root/管理员权限
    • 完全隔离

    虚拟机基于硬件虚拟化,而虚拟主机基于软件层或文件系统分割。

    所谓虚拟化,其本质就是资源的一个抽象话,将所有资源最小单元化,从而大大提高资源的利用率。虚拟化的管理程序Hypervisor(VMM),其位于虚拟机与底层硬件设备中间的虚拟曾,运行于硬件设备之上,负责对硬件资源进行抽象,为上层虚拟机提供运行环境所需的资源,并且使每一个虚拟机都能够互不干扰、相互独立地运行于同一设备当中。

    Docker

    docker是一种虚拟化技术,利用容器化的方式可以将应用程序及其依赖项打包成独立、轻量级别的环境,从而实现快速部署、高速运行和跨平台运行的功能。

    K8s

    Kubernetes(K8s),它是一个开源的容器编排平台,可以自动完成在部署、管理和扩展容器化应用过程当中涉及的许多手动操作。它提供了集群管理、服务发现、负载均衡等功能,使得用户可以更加轻松地管理容器化应用的生命周期。

    Docker与K8s

    由其我们可以看出Docker与K8s具有本质上的区别

    Docker:容器进行时

    • 负责创建和运行单个容器

    K8s:容器编排平台

    • 负责管理成百上千个容器

    从渗透测试角度出发

    • Docker:通常先进行容器逃逸,然后再进行内网的横向移动
    • K8s:通常先尝试接管集群,从而获取对容器和集群资源的完全控制

    虚拟主机

    Web漏洞 > /var/www/yoursite/ (受限的web应用)> 横向移动到同服务器的其它站点 > 提权到服务器

    虚拟主机主要攻击方式通过横向移动到其他网站、窃取敏感数据、植入后门等等,并不像虚拟机那样追求提权到root或者进行Hypervisor 逃逸

    虚拟机

    应用漏洞 > OS提权 > 利用Hypervisor漏洞 > 对宿主机进行控制

    Docker容器逃逸

    在渗透的时候,当我们拿到了一个shell后发现当前所处的环境是在docker容器里面,这个时候就需要我们进一步渗透,逃逸到宿主机当中,获取到宿主机的权限。因为当我们处于docker环境当中时对后续的渗透会有很多限制,如环境缺失、常用命令无法执行、网段不通等等。

    Docker逃逸

    容器的安全依赖于多层防护:

    容器安全= Namespace 隔离 + Cgroup 资源限制 + Capabilities 权限限制 + Seccomp 系统调用过滤 + AppArmor/SELinux 强制访问控制

    只要突破其中一层限制,就有可能进行逃逸!

    特权模式

    如果Docker容器是以 –privileged参数启动的话,Docker会:

    • 完全设备访问权限:可以访问宿主机的所有设备
    • 授权容器几乎所有的Linux Capavilities
    • 禁用安全隔离机制:包括Seccomp、AppArmor/SELinux限制
    • 允许容器挂载文件系统、允许容器加载内核模块
    • 它会有namespace隔离,但却很容易绕过
    • 虽然其禁用了很多,但特权容器和普通容器一样,都对Cgroup进行了限制

    Docker Socket逃逸

    在Docker容器当中,Docker Socket被用于与Docker守护进程(Docker daemon)进行通信。在前文当中我们也提到过Docker daemon,其主要是负责接收并处理来自 Docker Client 的请求,检查本地是否存在指定的镜像,如果没有的话会从Docker Registry当中拉取镜像,然后创建并启动容器。

    造成该漏洞的成因主要是容器在启动的时候将宿主机的Docker Socket挂载了进来。

    看完以上描述,我们应该可以理解Docker Socket逃逸,其主要是利用Docker Socket与daemon通信这一特征,然后在宿主当中创建一个挂载了宿主机目录的容器。

    Procfs危险挂载

    在linux中,/proc目录时一个伪文件系统,它里面暴露了内核的各种运行时候的参数和配置。并且其中的一些文件是可以写入的,写入后会影响宿主机内核的行为。

    如果在容器启动的时候宿主机的/proc挂在进来就会造成该漏洞。

    在该漏洞当中我们要利用 /proc/sys/kernel/core_pattern文件。

    该文件主要是负责进程崩溃时候的内存数据转储。就是当进程崩溃的时候产生core dump的时候内核用什么程序来负责处理该dump文件。但是如果内容是以**管道符’|’**开头的话,那么内核就会把管道符后面的部分以命令行的方式进行解析并运行。由于容器共享主机的内核,所以该命令就会以宿主机的权限来运行。

    首先判断是否挂载了宿主机的procfs

    使用该命令来查找当前容器当中的procfs,如果找到了两个core_pattern,那么该容器挂载了宿主机的procfs。其中第二个就是宿主机的procfs,/host/proc/sys/kernel/core_pattern

    在前面我们也提到了该漏洞的成因就是利用/proc/sys/kernel/core_pattern会在系统崩溃的时候自动运行,然后内核会将管道符后面的内容以命令行的方式运行。因此一旦程序发生崩溃就会自动的运行该脚本,进行反弹宿主的shell。

    Cgroup 配置错误

    Cgroup是Linux内核用来限制和管理进程资源的一种机制。其中有一个关键的功能release_agent

    release_agent的作用是当cgroup当中的最后一个进程退出的时候,内核会自动的执行release_agent所指定的程序,并且是以宿主机root权限进行执行。

    该漏洞的成因是:Cgroup被可写的方式被挂载到了容器当中。

    与Procfs危险挂载类似,将恶意脚本文件的绝对路径写入到release_agent当中,在cgroup最后一个进程退出之后他会读取里面那个路径,从而造成攻击。

    首先判断容器内是否挂载了Cgroup文件系统,并且是否为读写模式

    要先创建一个临时目录来挂载Cgroup文件系统

    开启release_agent通知,当conf子目录的任务进程被清空的时候会触发内核

    使用sed命令从/etc/mtab文件中找到容器在宿主机上的真实路径

    设置 release_agent 为恶意脚本

    然后在容器当中写入恶意的脚本

    最后将当前shell的PID写入/tmp/test/conf/cgroup.procs文件当中,意味着当前shell进程将被添加到cgroup当中,其它进程将会被清空只保留当前的shell进程

    **$$**代表当前shell进程的PID,写入 cgroup.procs 把它加入到cgroup当中,sleep 0执行完后进程退出,cgroup变空,内核触发release_agent,以宿主机root的身份来执行恶意脚本。
